Background

Kore.ai has built SearchAssist product to offer “conversational search” experiences for enterprises. The key value proposition was to offer ‘search first’ and ‘chat first’ experience and allow the enterprises to choose how to deploy the search. It was closely integrated with ‘XO Platform / Automation’ so that the customers can seamlessly deploy these two products together.

With the advent of LLM/GenAI capabilities, the Retrieval Augmented Generation (RAG) architecture has opened a new world of opportunities in the ‘search industry’. SearchAssist was able to quickly adopt this framework and has moved towards “answers” as the primary capability. .

Content Source

SearchAssist simplifies finding answers by bringing together different data types from various sources in an organization – like the company website, third-party apps, blogs, help centers, and product catalogs. It handles diverse content formats such as webpages, documents, and media files. To manage, update, or explore data in SearchAssist, users can go to the Sources tab. Data is periodically refreshed, ensuring answers are generated from recent content.

Recognizing the challenges posed by extensive websites with numerous pages and links, SearchAssist ensures that indexed data and information appear in user search results, eliminating the need for tedious manual navigation.

Overall, SearchAssist provides a user-friendly and comprehensive solution for organizations seeking effective web data management.

Chunk Extraction Models

Text Extraction Model

  1. User Configuration: Users can tailor the chunk extraction process by defining the chunk size, specifying the number of tokens within each chunk. Additionally, they have the option to set chunk overlap, enhancing flexibility.
  2. Techniques Employed: The Text Extraction Model incorporates a fusion of techniques from natural language processing (NLP) and machine learning. Rule or Pattern-Based Extraction Model

  1. Operational Approach: This model operates on a rule-based methodology, adhering to predefined rules or patterns for identifying and extracting particular segments or chunks of information from the text.
  2. Rule Configuration: Users can establish rules designed to recognize distinct patterns or structures within the text. These rules serve as the guiding principles for the model to accurately pinpoint and retrieve chunks based on the defined criteria.

Layout-Aware Extraction Model

  1. Consideration of Layout: The Layout-Aware Chunk Extraction model stands out by taking into account the specific layout or structure of a document during the information extraction process.
  2. Spatial Elements Considered: Unlike conventional models, this approach considers the spatial arrangement and organization of content within a document. Structural elements such as headings, paragraphs, lists, titles, text, images, and tables are scrutinized.
  3. Enhanced Significance: The model aims to identify and extract chunks that are not just semantically relevant but also contextually coherent, leveraging the spatial arrangement within the document. This strategy significantly enhances the accuracy and contextual significance of the extracted information.

Summary Extraction Model

  1. NLP-based Summarization: The Summary Extraction Model is specifically designed for natural language processing, focusing on generating concise and coherent summaries of longer texts.
  2. Analytical Process: This model analyzes the input text, identifying key information and extracting the most relevant content. The outcome is a condensed version that encapsulates the essential meaning of the original document.
  1. User Facilitation: The goal is to provide users with a shortened yet informative representation of the source material. This facilitates an easier grasp of the main points, making the information more accessible and comprehensible.

Query Rewriting

Query rewriting functionality

Query rewrite involves combination following techniques-

  1. Rephrasing query

  1. The Re-Phrase Query Generator refinines user queries for effective information retrieval.
  2. In this a retriever utilizes an LLM (Language Model) to process the user's natural language query and generate a revised query tailored for a vector store.
  3. It extracts essential information from the user's input, eliminating irrelevant details for the retrieval task.
  4. For Example, if the user initially asks, "Hi, I'm Lance. What are the approaches to Task Decomposition?" the Re-Phrase Query Generator would streamline the query to focus solely on the core information, resulting in a revised query such as "approaches to Task Decomposition." This method optimizes the query for more accurate and efficient retrieval of relevant information from the vector store.

  1. Rewriting query using chat history

  1. Query Rewriting is a concept that involves altering a user's current query based on their past interactions and query history. Instead of treating each query in isolation, this approach takes into account the context of previous questions. By understanding the user's history, the system can dynamically adjust the current query to better align with the user's intent and preferences. This enhances the relevance of the responses and improves the overall user experience by providing more personalized and contextually appropriate information.
  2. Queries can be rephrased using the past queries only or the conversation i.e. query and the answer. User can decide how the query should be rephrased using what
  3. User can also define the number of previous queries or conversations that sould be used for query rewriting
  4. See the below example-
  1. User Previous Queries:
  1. "What are popular tourist destinations in Europe?"
  2. "Tell me about travel restrictions in Asia."
  1. User Current Query:

"What are the best budget-friendly hotels near famous attractions in these regions?"

  1. Rewritten Query:

"What are budget-friendly hotel options near popular attractions in Europe and Asia?

  1. In this example, the system recognizes the user's historical interest in travel-related information. The current query, which seeks budget-friendly hotel options near attractions, is rewritten to specifically address the user's preferences based on their past interactions. This personalized query rewrite aims to provide more relevant and helpful information for the user's travel inquiries.

  1. Generate multiple queries for complex queries

  1. The Multiple Question Generator is a powerful query engine designed to handle complex user inquiries involving multiple data sources. It operates by deconstructing intricate user queries into sub-questions, each tailored to a specific data source.
  2. The generator follows guidelines to ensure specificity, relevance, and answerability of the sub-questions. Multiple sub-questions are generated for each query, offering a nuanced approach to address different aspects of the user's overarching question.
  3. For example, given a user query like "Explore historical and current trends in renewable energy companies," the Sub-Question Generator might generate sub-questions such as 'Provide historical performance metrics for SolarCo' and 'What are the recent innovations in WindPowerTech?' These sub-questions are linked to the corresponding conext, allowing for a systematic extraction of relevant information from different data sources and facilitating a comprehensive response to the user's inquiry.

Re-ranking

Overview

  • Purpose: Enhances the quality of retrieved results by reordering them based on refined relevance assessments.
  • Position: It follows initial retrieval, acting as a final precision filter before downstream tasks like answer generation.
  • Goal: Ensures the most relevant chunks are presented at the top, optimizing user experience and downstream task performance.

Cross-Encoders for Re-ranking

Functionality

  • Transformer-based deep learning model, often fine-tuned for semantic matching or sentence similarity tasks.
  • Takes a pair of text inputs (query and retrieved chunk) as a single sequence, separated by a special delimiter token.
  • Processes the combined sequence, attending to both query and chunk simultaneously.
  • Outputs a single score between 0 and 1 representing the predicted relevance of the chunk to the query.
  • Re-ranking flow
  1. Initial Retrieval: A base retriever (e.g., dense passage retrieval) fetches a set of candidate chunks for a given query.
  2. Re-ranking:
  1. Each query-chunk pair is fed into the Cross-Encoder.
  2. The Cross-Encoder assigns a relevance score to each chunk.
  3. Chunks are reordered based on these scores, prioritizing the most relevant.
  1. Top-k Selection: The highest-scoring k chunks are sent to LLM for answer generation

Advantages

  • Captures semantic relationships: Assesses relevance beyond keyword matching, considering deeper semantic nuances and context.
  • Handles diverse data types: Versatile for text, code, or other structured data, adaptable to various retrieval tasks.
  • Trainable: Can be fine-tuned for specific domains or tasks to improve relevance assessment.

GPT Cache

GPT Cache aims at enhancing language model performance and efficiency by seamlessly integrating advanced caching mechanisms. Its primary objective is to optimize the retrieval process of relevant information by strategically storing precomputed embeddings and their corresponding similar vectors.

Cache Storage: Organizing for Rapid Retrieval

The Cache Storage component acts as the central repository for vectors and their associated similar vectors. This critical element stores these pairs as key-value entries, meticulously arranging them in descending order based on their distance or similarity. This organizational strategy enables the system to swiftly retrieve the most relevant and similar vectors during the processing of user queries, contributing to an expedited and optimized retrieval process.

Cache Retrieval: Vector Existence Checking

In the query processing phase, the Cache Retrieval functionality comes into play. This feature is designed to determine whether a given vector already exists in the cache storage. By proactively checking for vector existence, the system can efficiently identify and retrieve previously stored vectors, potentially eliminating the need for redundant computations. This proactive approach further enhances the overall efficiency of the system.

Evaluation metrics

  • This evaluation framework is specifically designed to assess Answers generated through Retrieval Augmented Generation (RAG) pipelines, which seamlessly integrate retrieval models and generative language models (LLMs).
  • By combining these models, RAG pipelines aim to deliver more accurate and comprehensive answers to intricate questions. The framework introduces four crucial metrics for evaluating the performance of different components and gauging the overall effectiveness of the pipeline:

  • Context Precision

This metric gauges the signal-to-noise ratio within the retrieved context. It calculates the proportion of pertinent information compared to irrelevant details. A high context precision indicates the retriever's adeptness at selecting the most relevant information for the LLM to utilize.

  • Context Recall

Focusing on the completeness of the retrieved context, this metric measures the percentage of relevant information successfully brought to the LLM from the entire knowledge source. A high context recall suggests that the retriever captures all significant information, ensuring the generation of a comprehensive and accurate answer.

  • Faithfulness

Evaluating the factual accuracy and adherence to the retrieved context, this metric compares the generated answer to the retrieved information. It assesses how faithfully the answer reflects the provided sources. A high faithfulness score indicates that the LLM avoids introducing misleading or irrelevant information into the response.

  • Answer Relevance

This metric assesses the relevance of the generated answer to the original question. It evaluates how effectively the answer addresses the user's query and fulfills its intended purpose. A high answer relevance score signifies that the LLM comprehends the question's intent, providing a meaningful and satisfactory response.

  • These four metrics collectively offer a comprehensive evaluation of answer performance. By scrutinizing each metric individually and considering their combined insights, developers and users can gain a nuanced understanding of their system's strengths and weaknesses, facilitating targeted improvements.

  • Benefits and Advantages:

  • Comprehensive Assessment: The framework offers a holistic approach to evaluating the performance of RAG pipelines. By employing multiple metrics, including Context Precision, Context Recall, Faithfulness, and Answer Relevance, it provides a thorough and nuanced assessment of different aspects of answer generation.
  • Targeted Improvement: Developers and users can leverage the individual insights from each metric to identify specific strengths and weaknesses within their system. This targeted information allows for focused efforts in refining and enhancing the components that may be underperforming.
  • Optimized Retrieval: The Context Precision and Context Recall metrics specifically focus on the retrieval aspect of RAG pipelines. High scores in these metrics indicate an effective selection of relevant information and completeness of the retrieved context, optimizing the performance of the retrieval models within the pipeline.
  • Factual Accuracy: The Faithfulness metric ensures that the generated answers accurately reflect the information retrieved from knowledge sources. This emphasis on factual accuracy is crucial for building user trust and ensuring that the responses align with the provided sources.

Guardrails for Answers

Overview

Guardrails ensure the responsible use of Language Models (LLMs) and provide accurate responses, fact-checking, safety, and topical guardrails collectively contribute to the ethical deployment of LLMs. These guardrails play a pivotal role in the development process, fostering responsible utilization of language models across various applications.

Fact-checking Guardrail

The fact-checking guardrail is a crucial component of ensuring the accuracy and reliability of answers generated by Language Models (LLMs). Its primary purpose is to verify the factual correctness of information provided by the LLM. This guardrail involves cross-referencing generated content against trusted sources or databases to identify and rectify any inaccuracies or false claims. By implementing fact-checking guardrails, developers aim to enhance the credibility and trustworthiness of the LLM's responses.

Function

Cross-referencing generated content against trusted sources to identify and rectify factual errors or false claims.

Example

An LLM claims that the capital of France is London. The fact-checking guardrail would verify this information and correct it to Paris.

Bias, Harmfulness, and Toxicity Check

Bias

Definition: Bias refers to answers with a partiality or inclination towards a specific viewpoint, group, or set of values, often resulting in an unfair or unbalanced representation.

Example: An LLM asked to provide information about different cultures uses biased language describing certain groups as "primitive" or "backward."

Harmfulness

Definition: Harmfulness signifies answeer with a potential to cause physical, emotional, or psychological harm to individuals or entities.

Example: An LLM responding to a query about weight loss suggests dangerous methods like extreme fasting or unhealthy diet pills.

Toxicity

Definition: Toxicity involves LLM using offensive language or spreading hate speech. Toxicity guardrails keep such negativity at bay.

Example: An LLM responding to a user's emotional expression with dismissive or mocking language.

Safety Guardrails

Safety guardrails serve as a proactive mechanism to filter out answers that may pose risks to users or violate safety standards. These guardrails are designed to identify and prevent the generation of answers that could be considered harmful, inappropriate, or unsafe.

Topical Guardrails

Topical guardrails are implemented to confine the LLM's responses within predefined topics or conext. By doing so, users aim to enhance the specificity, accuracy, and relevance of the generated answers.

Function

Confining the LLM's responses to predefined topics or context to ensure specificity and accuracy.

Example

A user asks about scientific advancements in renewable energy. The topical guardrail would prevent the LLM from rambling about politics or unrelated technologies.

Key features of topical guardrails include:

Topic Relevance

Ensuring that the LLM provides answers that align with the intended subject matter.

Boundary Enforcement

Restricting the generation of responses outside the specified topics, preventing the LLM from straying into irrelevant or off-topic territory.

Enhanced Precision

Improving the overall precision and focus of the LLM by limiting its scope to specific domains or context.
